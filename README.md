# scaffor

### **Deterministic scaffolding for AI agents. No planning. No drift.**

Your LLM shouldn't invent your file structure.

scaffor moves scaffolding out of the agent's head and into an **executable contract**:
same inputs → same files → same structure.

**One command. Same files. Every time.**

## The problem nobody admits

Ask your agent to "add a new entity to the hexagonal app". Watch what happens:

1. It reads three files to *guess* your conventions
2. It plans the file tree — sometimes correctly
3. It writes `domain/User.go`, then notices the rest is lowercase and rewrites
4. It forgets the repository interface, you remind it
5. It wires the handler in the wrong `init.go`
6. You re-prompt. It drifts further from the existing style.

Every one of those steps costs tokens, time, and context. Multiply by every file in the feature and you see the bill. And the worst part isn't the cost — it's that **the output is non-deterministic**. Two agents, two sessions, two different layouts.

## The fix

```bash
scaffor execute hexagonal-cli add_entity --set Entity=User
```

```
✅ Created:
  internal/myapp/domain/user.go
  internal/myapp/domain/repositories/user/user.go
  internal/myapp/outbound/user/user.go
  internal/myapp/app/commands/user_handler.go

→ NEXT STEPS:
  1. Define User fields in domain/user.go
  2. Implement the adapter in outbound/user/
  3. Wire the handler in init.go
```

Files: **deterministic**. Naming: **enforced**. Hints: **machine-readable next actions**.
The agent stops planning and starts *filling in the blanks*.

## Why scaffor

### 1. Structure becomes a contract, not a guess

Templates are YAML manifests: inputs declared, outputs declared, paths declared. Same inputs → same files → same structure. Every time. Across agents, across sessions, across team members.

### 2. Hints replace planning

Every command prints a **structured hint** — the template author's designed workflow for what comes next. The agent doesn't brainstorm; it reads the hint and executes. No drift, no improvisation, no "let me also add…". This is **progressive disclosure applied to workflows**: the next action arrives exactly when it's needed, not as upfront context bloat.

### 3. Small models suddenly become enough

When the agent no longer has to plan file trees and remember conventions, **the reasoning burden collapses from O(n files × m conventions) to O(1)**. In practice that means **Haiku handles tasks that previously needed Sonnet or Opus** — not because it got smarter, but because you stopped asking it to reason about things a template can just *know*. Fewer tokens in context, fewer decision points per turn, fewer opportunities to drift.

> All examples in [`/examples/`](examples) were generated by Claude Haiku with only `scaffor --help` as context. No custom prompts. No skills. No prompt engineering.

### 4. Validated before use

Templates are **linted statically** (`scaffor lint`) and **tested end-to-end in a sandbox** (`scaffor test`). Broken templates fail in CI, not in your agent's next turn.

---

## Where scaffor fits

If you've read the recent literature on agent architectures, scaffor is an opinionated implementation of three converging ideas:

- **Progressive disclosure** (Anthropic Skills, MCP instructions): agents receive context just-in-time, not upfront
- **Deterministic tool use** over free-form generation: reduce the agent's decision space where the correct answer is mechanical
- **Structured success and error channels** (MCP): the agent's next action is machine-readable, not inferred from prose

The combination produces a scaffolding engine where the agent plans less, hallucinates less, and can fall back to smaller models without losing reliability.

## Quick Start

```bash
# Install (userspace, no root)
curl -sSL https://raw.githubusercontent.com/JLugagne/scaffor/main/install.sh | sh

# Discover
scaffor list
scaffor doc hexagonal-cli bootstrap

# Execute
scaffor execute hexagonal-cli bootstrap \
  --set AppName=myapp \
  --set ModulePath=github.com/org/myapp
```

Make your agent scaffor-aware: copy [`AI_INSTRUCTIONS.md`](AI_INSTRUCTIONS.md) into your `.cursorrules`, `.clinerules`, `AGENTS.md`, or system prompt. The agent will then always `list` before writing files manually.

## Configuration

scaffor reads a global config at `~/.config/scaffor/config.yml` (or `$XDG_CONFIG_HOME/scaffor/config.yml`) that declares directories it should scan for templates — so you don't have to copy templates into every project or pass `--templates-dir` each time.

```bash
# Write a commented example config
scaffor init-config

# Open the config in your editor ($EDITOR, falling back to $VISUAL)
scaffor edit-config

# Or edit it directly:
cat > ~/.config/scaffor/config.yml <<EOF
template_sources:
  - path: ~/work/scaffor-templates
    description: Personal templates
  - path: ~/work/team-scaffor-templates
    description: Shared team templates
EOF

# Verify scaffor sees them
scaffor config
```

Rules:
- Paths support `~`, `$VAR`, and `${VAR}`. Relative paths are rejected.
- First match wins. Duplicate template names across sources emit a stderr warning.
- `--templates-dir <path>` overrides the config for a single invocation.
- `--ignore-missing-sources` tolerates source directories that don't exist on disk (useful when one of your repos is temporarily unavailable).

When no config is present, scaffor falls back to `.scaffor-templates/` in the current directory — unchanged behavior.

## 🔌 MCP Server

scaffor ships a built-in [Model Context Protocol](https://modelcontextprotocol.io) server. Claude Code, Cursor, and any MCP client drive scaffolding **natively** — no CLI parsing, no shell heuristics. Tools return **structured content** (`dry_run`, `skip`, `force` flags, typed errors) rather than free-text, letting agents retry programmatically on failure instead of string-matching their own output.

```bash
scaffor mcp
```

Six tools over stdio:

| Tool | Purpose |
|---|---|
| `list` | Enumerate templates from all configured sources (or `.scaffor-templates/` if no config) |
| `doc` | Show docs for a template or command (`all=true` dumps every variable) |
| `execute` | Run a command with variables. Supports `dry_run`, `skip`, `force` |
| `lint` | Validate a template manifest |
| `test` | Run the template's test block in a temp directory |
| `status` | Review the session log of tool calls and file events |

Every session is logged to `.scaffor/<session-id>.jsonl` — **full auditability** of what was created, overwritten, or skipped. Your agent can review its own work.

<details>
<summary><b>Configure for Claude Code / Cursor</b></summary>

```json
{
  "mcpServers": {
    "scaffor": {
      "command": "scaffor",
      "args": ["mcp"]
    }
  }
}
```

The server auto-advertises instructions telling the client to always `list` before generating files manually.
</details>

## Templates

A template is a directory with a `manifest.yaml`. Commands declare variables, files, post-commands, and hints:

```yaml
name: hexagonal-cli
commands:
  - command: add_entity
    description: Add a new domain entity with its repository and adapter
    variables:
      - key: Entity
        description: Entity name in PascalCase
    files:
      - source: domain/entity.go.tmpl
        destination: internal/{{ .AppName }}/domain/{{ .Entity | lower }}.go
      - source: domain/repositories/entity/entity.go.tmpl
        destination: internal/{{ .AppName }}/domain/repositories/{{ .Entity | lower }}/{{ .Entity | lower }}.go
    post_commands:
      - command: "goimports -w {{ .Files }}"
        mode: all
    hint: |
      Entity {{ .Entity }} added.
      Now:
        1. Fill in the fields in domain/{{ .Entity | lower }}.go
        2. Implement the adapter
        3. Wire the handler: scaffor execute hexagonal-cli add_command --set Command={{ .Entity | lower }}
```

**Composable:** chain commands, emit hints that reference the next one.
**Language-agnostic:** Go, Python, TypeScript, Terraform, docs — any text file.
**Go-template based:** with `lower`, `upper`, `title` helpers.

### Post-command modes

| Mode | Behavior |
|---|---|
| `all` *(default)* | Runs once after all files, with `{{ .Files }}` = space-joined paths |
| `per-file` | Runs once per created file, with `{{ .File }}` = that file's path |

By default, post-commands are **printed for review, not executed**. Pass `--run-commands` to run them automatically.

## Linted statically, tested in sandbox

### `scaffor lint` — shift template errors left

```
$ scaffor lint broken-template
LINT ERRORS in broken-template:

  command "create", field "files.destination":
    variable "Nme" used but not declared (did you mean "Name"?)
  command "create", field "post_commands":
    post_command[0] has an empty command

2 issue(s) found
```

Detects undeclared variables, broken references, invalid modes. Suggests fixes via Levenshtein distance. Runs in milliseconds — **put it in CI** so broken templates fail before they reach an agent's turn.

### `scaffor test` — validate templates end-to-end

```yaml
test:
  - command: bootstrap
    params: { AppName: testapp, ModulePath: github.com/test/testapp }
  - command: add_entity
    params: { AppName: testapp, ModulePath: github.com/test/testapp, Entity: Book }

validate:
  - "go mod tidy"
  - "go build ./..."
```

Runs the full scenario in a throwaway directory. Executes validation commands. Reports coverage (`N of M commands exercised`). Non-zero exit = failed template.

**Catch regressions before they hit a real project.**

## 🔒 Safety

> ⚠️ **Templates can execute arbitrary shell commands.** Review any template you did not author. Use `--dry-run` to preview.

**Threat model**: scaffor trusts the local template source. Your agent invokes scaffor, scaffor invokes your shell. Pin your sources in the global config and review manifests like you review dependencies.

- **Shell commands are printed, not executed**, unless you pass `--run-commands`
- **Pre-flight checks** abort if any destination file already exists
- **Per-file overrides** via `on_conflict`: `default` (respects flags), `skip` (always skip), `force` (always overwrite)
- **Directory traversal** (`..`) in destinations is rejected
- **Session log** at `.scaffor/<session-id>.jsonl` records every file event — auditable supply-chain trail for every generated file
- **Treat templates like code** — only use sources you trust

## Installation

All install paths below are **userspace** — no `sudo`, no system directories touched.

```bash
# Recommended: one-liner (Linux, macOS, WSL). Installs to ~/.local/bin.
curl -sSL https://raw.githubusercontent.com/JLugagne/scaffor/main/install.sh | sh

# Or pick a specific version / install directory:
curl -sSL https://raw.githubusercontent.com/JLugagne/scaffor/main/install.sh \
  | VERSION=v0.5.0 BINDIR=$HOME/bin sh

# Via Go (builds from source, installs to $GOBIN or ~/go/bin)
go install github.com/JLugagne/scaffor/cmd/scaffor@latest

# From source
git clone https://github.com/JLugagne/scaffor.git
cd scaffor && go build -o scaffor ./cmd/scaffor

# Shell completion (userspace locations)
scaffor completion bash > ~/.local/share/bash-completion/completions/scaffor
scaffor completion zsh  > "${fpath[1]}/_scaffor"
scaffor completion fish > ~/.config/fish/completions/scaffor.fish
```

If `~/.local/bin` is not on your `$PATH`, the installer prints the exact line to add to your shell rc.
